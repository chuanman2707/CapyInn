# Release signing

CapyInn ships updater releases through [`.github/workflows/release.yml`](../.github/workflows/release.yml).

The workflow now runs in four stages:

1. `verify-version` checks that the tag matches the version in `mhm/package.json`, `mhm/src-tauri/tauri.conf.json`, and `mhm/src-tauri/Cargo.toml`.
2. `verify-app` runs the shared validation pass before any release artifacts are built.
3. `build-windows`, `build-macos-aarch64`, and `build-macos-x86_64` each build one platform and emit per-platform metadata with the asset name and embedded updater signature.
4. `publish-release` downloads every build artifact, generates the canonical `latest.json`, verifies the full release payload exists, and only then creates the GitHub Release.

## Release trigger

Push a version tag that matches all three version sources:

```bash
git tag v0.1.0
git push origin v0.1.0
```

If the tag does not match the checked-in app version, the workflow fails before any release job runs.

## GitHub settings

Go to `Settings -> Secrets and variables -> Actions`.

### Required secrets

- `TAURI_SIGNING_PRIVATE_KEY`: private key content generated by `tauri signer generate`. This signs updater artifacts and is required for release builds.
- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`: password for the updater private key if the key is encrypted.

### Required variables

- `TAURI_UPDATER_PUBLIC_KEY`: the matching updater public key. The workflow injects this into a generated `mhm/src-tauri/tauri.release.conf.json` and does not store it in the checked-in `tauri.conf.json`.

### Optional Windows signing secrets

- `WINDOWS_CERTIFICATE`: base64-encoded `.pfx` certificate for signing the Windows NSIS installer.
- `WINDOWS_CERTIFICATE_PASSWORD`: password for the `.pfx` export.

If the Windows certificate secrets are absent, the workflow still builds the Windows updater bundle, but the NSIS installer is unsigned.

### Optional variables

- `WINDOWS_TIMESTAMP_URL`: RFC3161/AuthentiCode timestamp server URL. If absent, the workflow falls back to `http://timestamp.digicert.com`.

### Workflow permissions

In `Settings -> Actions -> General -> Workflow permissions`, enable `Read and write permissions` so the final publish job can create the GitHub Release.

## Preparing secrets

### Export the Windows certificate

On Windows:

```powershell
certutil -encode certificate.pfx base64cert.txt
```

Copy the base64 payload from `base64cert.txt` into `WINDOWS_CERTIFICATE`.

### Generate updater signing keys

From `mhm/`:

```bash
npm run tauri signer generate -- -w ~/.tauri/capyinn.key
```

Store the private key contents as `TAURI_SIGNING_PRIVATE_KEY`.

The `generate` command prints the matching public key when it creates the keypair. Copy that printed public key into the repository variable:

```bash
npm run tauri signer generate -- -w ~/.tauri/capyinn.key
```

Set the printed public key value as `TAURI_UPDATER_PUBLIC_KEY`.

## Release shape

Pushing `vX.Y.Z` produces a single GitHub Release containing:

- a Windows NSIS updater package and its `.sig`
- a macOS Apple Silicon `.app.tar.gz` updater package and its `.sig`
- a macOS Intel `.app.tar.gz` updater package and its `.sig`
- one canonical `latest.json`

The generated `latest.json` keeps an explicit platform contract:

- `windows-x86_64`
- `darwin-aarch64`
- `darwin-x86_64`

Each manifest entry uses an immutable asset URL in the form `https://github.com/<owner>/<repo>/releases/download/vX.Y.Z/<asset-name>`. Mutable `releases/latest/...` URLs are intentionally rejected by the generator.

## Generated release config

The workflow creates `mhm/src-tauri/tauri.release.conf.json` inside CI for each platform build. That generated config:

- enables `bundle.createUpdaterArtifacts`
- injects `plugins.updater.pubkey` from `TAURI_UPDATER_PUBLIC_KEY`
- adds Windows signing metadata when a certificate thumbprint is available

Nothing in the checked-in Tauri config needs to carry release-only signing state.

## Runtime behavior

- Windows: the app downloads the update, shows `Restart to update / Later`, and exits into the installer flow when the user confirms.
- macOS: the app downloads the update, shows `Restart to update / Later`, and installs plus relaunches when the user confirms.

## What the workflow does

1. Verifies version alignment across package, Tauri, Cargo, and tag metadata.
2. Runs the shared verification pass before any release build starts.
3. Builds Windows, macOS Apple Silicon, and macOS Intel updater artifacts in isolated jobs.
4. Collects per-platform metadata so the final publish job can assemble immutable manifest URLs.
5. Generates `latest.json` with `mhm/scripts/generate-latest-json.mjs`.
6. Creates the GitHub Release only after every asset and `latest.json` are present.
